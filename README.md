> **2026-07-17 追記㉗：ホーム↔感知器・写真・物件情報のスクロール連動を実機に合わせて再対策
> （慣性スクロールを一時的にロックする方式に変更）**
>
> 追記㉖でのスクロール対策（documentElement/bodyのscrollTopリセット＋描画フレームでの
> 二重リセット）について、「感知器・写真・物件情報タブが、常にトップからではなく、直前の
> ホーム画面のスクロール位置に近い中途半端な位置から表示される」というご指摘とスクリーン
> ショットをいただきました。ご指摘のとおり、スマートフォン実機（特にiOS Safari）では
> フリック操作後の慣性スクロールが数百ミリ秒〜1秒近く残ることがあり、描画フレーム単位
> （数十ミリ秒）でのリセットでは追いつかず、タブ切替後もその慣性スクロールに引きずられて
> しまうケースがあることが分かりました。
>
> 今回は、タブ切替の瞬間だけ`html`・`body`の`overflow`を一時的に`hidden`にしてスクロール
> そのものを強制的に停止する方式に変更しました。これにより、切替前に残っていた慣性スクロール
> も含めてその場で完全に打ち切られます。その間にスクロール位置を0へ確定させ、慣性が収まる
> のに十分な時間（260ms）が経ってからスクロールを再度有効にします。
>
> なお、実装の過程で「タブを素早く連続でタップすると、スクロールロックの解除処理が
> 正しく行われず、画面がスクロールできなくなったまま固まってしまう」という不具合を
> テストで発見し、あわせて修正しました（複数の解除タイマーが重なった際、常に最後の
> 切替のタイマーだけが解除処理を行うように修正）。

> **2026-07-17 追記③：オフライン対応（通信断でのデータ消失バグの修正）**
>
> コードを確認したところ、「オフライン対応が無い」という以上に、一時的な通信断で

> 通知も無く、画面表示用のデータもページのメモリ上にしか無かった（reloadで消える）

>
> - **`index.html`を更新**：部屋の点検結果・設備点検・タグ・メモなど全ての保存処理を、
>   IndexedDB（ブラウザ内のローカルデータベース）に即時保存してから送信するように
>   変更しました。送信が失敗した場合は「送信キュー」に積まれ、通信が復活すると自動的に
>   再送されます。「本日の進捗」の右に「未送信N件」というバッジが表示され、今の同期状況が
>   一目でわかるようにしました。写真（base64）も含めてキャッシュ対象にしているため、
>   localStorageではなく容量に余裕のあるIndexedDBを使っています。
> - **`supabase-integration.js`を更新（重要な既存バグの修正）**：ログイン直後の内部処理が
>   「今日の点検セッションの準備」というネットワーク呼び出しの完了を待ってから
>   `window.storage`を使えるようにしていたため、オフライン時にはこの待ちがいつまでも
>   終わらず、上記のローカルキャッシュの仕組み自体が一切働かないという問題がありました。
>   `window.storage`を先に使えるようにし、点検セッションの準備は裏側で自動的に
>   リトライするよう修正しています。
> - **副次的に発見・修正した点**：静的HTMLのプレースホルダーに実在しそうな具体的な値
>   （建物名「コスモ宝塚シティフォルム」・件数「37件」）がハードコードされており、
>   何らかの理由でJavaScriptが実行されない場合（例：iOSの「クイックルック」のような
>   JS実行が制限されたプレビュー環境で開いた場合）に、見た目はそれらしいが実際には
>   読み込みが完了していない画面になってしまう問題がありました。明らかに「未読み込み」と
>   分かる表示（「読み込み中…」等）に変更しています。

>   `supabase-integration.js`の差し替えだけで反映されます）。

>   自動再送を含む）、および `supabase-integration.js` 単体の順序検証テスト6件で動作確認済みです。

> 勝手に参加できてしまう、複数の顧客企業へ販売する（商品化する）上で重大な情報漏洩
> リスクでした。今回、これを塞ぎ、**管理者(admin)が発行した招待リンクを持っている人だけが

>   なるようにし、`property_members` への直接insertを塞いでいます。今回も、既存のテーブルを

## デプロイ手順（Vercelを初めて使う場合）

### 1. Anthropic APIキーを取得する

1. https://platform.claude.com/settings/keys を開く（旧URLの
   console.anthropic.com からもここに転送されます）
2. 「Create Key」のようなボタンから新しいAPIキーを発行する
3. 表示されたキー（`sk-ant-...`で始まる文字列）をコピーしておく
   （このキーは二度と全文表示されないことが多いので、必ずこの時点で控えてください）

### 2. このフォルダをGitHubリポジトリにする

```bash
cd (このフォルダ)
git init
git add .
git commit -m "FireFlow Live Board: Vercel対応"
```
GitHub上で新規リポジトリを作成し、`git remote add origin ...` → `git push` で
アップロードしてください。

### 3. Vercelにインポートする

1. https://vercel.com でアカウント作成・ログイン
2. 「Add New...」→「Project」→ 先ほどのGitHubリポジトリを選択してインポート
3. フレームワークは自動検出で問題ありません（「Other」のままでOK）。ビルド設定も
   変更不要です（静的HTML + `/api`配下のFunctionsをそのまま認識します）

### 4. 環境変数を設定する

1. プロジェクト → **Settings → Environment Variables**
2. `ANTHROPIC_API_KEY` という名前で、手順1で控えたキーを値として追加
3. 環境は Production / Preview / Development すべてにチェック

### 5. デプロイする

初回インポート時に自動でデプロイが走ります。環境変数を後から追加した場合は、
**Deployments** タブから「Redeploy」を実行してください（環境変数の変更は
再デプロイしないと反映されません）。

デプロイが終わると `https://(プロジェクト名).vercel.app` のようなURLで
Live Boardにアクセスできるようになります。

### 6. 動作確認

1. 発行されたURLを開き、いつも通りログインする
2. 「点検希望時間連絡票」の画面から、実際の連絡票を1枚（またはPDF一式）読み込ませる
3. 部屋番号・区分・時刻に加えて、終了時刻／手書きメモ／お名前が正しく読み取れて
   いるか確認する
4. うまく読み取れない場合は、ブラウザの開発者ツール（コンソール）にエラーが出て
   いないか、Vercelダッシュボードの **Deployments → Functions** のログに
   `api/scan-time-request` のエラーが出ていないか確認してください
   （多くの場合、環境変数の設定漏れか再デプロイ忘れが原因です）。

## 実行時間について

Vercelの関数の実行時間上限は、Hobby（無料）プランでもデフォルト300秒（5分）
です（2026年7月時点の公式ドキュメント確認済み）。65ページのPDF一括スキャンでも
通常はこの範囲内に収まりますが、念のため `vercel.json` で `api/*.js` に対して
明示的に120秒を設定しています。それでも間に合わない場合は `vercel.json` の
数値を大きくしてください（Hobbyプランの上限は300秒です）。
